# Data

## Goal

Turn data into a traceable, reproducible, versioned training asset. This phase may write lightweight manifests or download scripts, but heavy conversion belongs to `preprocess`.

## Artifact Location

Read and write phase artifacts under `.claude/vibe-training/runs/<run_id>/`. If no run directory is known, find the latest project-local vibe-training run or create one. Do not write to `~/.claude/plans`.

## Inputs

- `plan.md`
- `model-choice.md`
- Raw data sources
- Annotation rules
- Privacy and license constraints

## Outputs

Write or update `dataset-spec.yaml` with:

- data sources
- license and privacy notes
- dataset hub/auth requirements, if any
- schema
- cleaning rules
- secondary alignment rules, if raw data labels/format/objective do not match the training target
- filtering rules
- data creation pipeline, if no suitable public dataset exists
- synthetic data plan
- split policy
- versioning convention

If implementation is requested, also create or patch the smallest useful source/download script or manifest. Do not stop at prose when a runnable data command is clearly needed.

## What To Do

1. Define source data and license boundaries.
2. Search Hugging Face Datasets first when public data is acceptable, then check official benchmarks, paper-linked datasets, Kaggle, domain repositories, or a custom data creation pipeline.
3. Plan download, sampling, and storage.
4. Plan cleaning, deduplication, filtering, redaction, and secondary alignment when raw data does not match the target exactly.
5. Define synthetic data generation only when it improves coverage or controllability.
6. Define train, validation, and test splits with leakage checks.
7. Record dataset version and reproducibility steps.
8. For executable work, run a tiny source check or dry run before any large download.

## Public Dataset Search Priority

When public data is allowed, search in this order:

1. Hugging Face Datasets and dataset cards
2. official benchmark/task pages
3. paper-linked datasets and project pages
4. Kaggle or domain repositories
5. general web search
6. custom collection, labeling, or generation pipeline

For Hugging Face datasets, record dataset id, config/subset, split names, dataset card URL, license, revision or commit when available, size, schema/features, known caveats, and whether streaming mode is useful.

## Hub Downloads And Access

For Hugging Face datasets, Kaggle, cloud buckets, private corpora, or other gated sources:

- record dataset id, version/revision, license, expected size, cache path, and required env var names
- do not record secret values
- check license/auth before large downloads
- plan resumable downloads or mirrors for large files and unstable networks
- if download or extraction fails, preserve the exact source id/path, partial cache location, error, and retry/fallback decision

## Cleaning And Secondary Alignment

If necessary, define a second alignment pass before training:

- clean corrupt, empty, duplicated, low-quality, policy-violating, or out-of-distribution records
- normalize labels, units, languages, encodings, timestamps, coordinate systems, image/audio formats, or text fields
- remap task schema when source labels do not exactly match the target label space or objective
- align instruction/response, prompt/completion, image/text, audio/text, ranking pairs, trajectories, or multimodal fields to the selected trainer contract
- define confidence thresholds, human review rules, automatic validators, and reject/repair policies
- keep raw, cleaned, and aligned dataset versions separate with counts and checksums
- verify that cleaning/alignment rules do not leak validation/test answers or remove hard-but-valid examples

## No Suitable Public Dataset

If no suitable public dataset exists, design a data creation pipeline instead of forcing a weak dataset match:

- define collection sources, labeling interface, annotation rubric, reviewer policy, and quality gates
- define synthetic generation prompts/templates/simulators only when they match the target distribution or cover known gaps
- include human review or automatic validators for generated labels
- create seed examples, negative cases, edge cases, and a holdout set that is not generated by the same pipeline when possible
- record provenance, license/privacy constraints, deduplication rules, and leakage checks
- run a tiny end-to-end sample before scaling collection or generation

## AIGC Generation Pipeline

When using AIGC or API-based generation for data creation:

- define whether generation uses local models, hosted APIs, internal services, or human-in-the-loop editing
- record provider/model names, API endpoint type, env var names for credentials, rate limits, expected token/image/audio/video units, and cache/retry policy; never record secret values
- estimate generation cost and wall-clock time before scaling: prompts count, average input/output tokens or media units, retries, validation calls, and failed-sample allowance
- run a small pilot batch and measure quality, latency, token usage, rejection rate, and post-filter yield before full generation
- define safety filters: privacy/PII removal, copyrighted or licensed source avoidance, harmful content filtering, prompt injection defense for source documents, and leakage prevention from validation/test sets
- define quality controls: schema validation, label consistency checks, diversity checks, deduplication, adversarial/edge-case coverage, human audit sampling, and automatic judge limitations
- keep generated data provenance: prompt template version, model version, seed/input sources, timestamp, filter version, and acceptance reason
- avoid training only on synthetic data unless the target explicitly tolerates synthetic distribution shift; preserve real or human-reviewed holdout data when possible

## Question Policy

Use `AskUserQuestion` when data decisions require user authority: whether private data may be used, license tolerance, labeling budget, synthetic/AIGC data acceptance, API/token budget, generation duration, provider/account choice, privacy/redaction rules, class balance priorities, test-set ownership, whether external datasets may be downloaded, and whether a new data creation pipeline is acceptable. Do not ask about facts you can inspect from existing files.

## Do Not Do

- Do not change training code.
- Do not deploy to GPU servers.
- Do not make final evaluation claims.
